## Overview

The `SOPTemplate` OAPP transforms Standard Operating Procedures from static documents into **living holonic workflows** — executable via the CRE runtime (`api/workflow`), AI-guided by BRAID (`api/braid`), Avatar-signed for accountability, and published to STARNET for cross-organisation sharing.

### The 5 Zomes at a Glance

| Zome | Purpose | Key Holons |
|------|---------|------------|
| **SOPZome** | Procedure definitions + version history | `SOPHolon`, `SOPVersionHolon` |
| **StepZome** | Instructions + branching logic | `SOPStepHolon`, `SOPDecisionPointHolon`, `SOPConditionBranchHolon` |
| **RunZome** | Execution instances + evidence | `SOPRunHolon`, `StepCompletionHolon`, `SOPDeviationHolon` |
| **AuditZome** | Audit certificates + analytics | `SOPAuditHolon`, `SOPAnalyticsHolon` |
| **IntegrationZome** | External system connections | `IntegrationTriggerHolon`, `WebhookConfigHolon`, `ExternalSystemLinkHolon` |

**Total holons:** 13  
**Total zomes:** 5

## SOPZome

### SOPHolon

The master definition of a procedure. Every other holon in the template points back here via `SOPId`.

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `Name` | string | Human title, e.g. "Enterprise Customer Onboarding v3" |
| `Version` | string | Semantic version. Incremented on approved changes. |
| `Status` | Enum | `Draft` → `Active` → `Deprecated` → `Archived` |
| `OwnerAvatarId` | Guid | Accountable Avatar for this SOP |
| `RequiredRoles` | string[] | e.g. `["CustomerSuccessManager", "SolutionsEngineer"]` |
| `EstimatedDurationMinutes` | int | Expected wall-clock completion time |
| `Category` | Enum | `CustomerSuccess`, `Operations`, `HR`, `Compliance`, `Finance`, `Sales`, `Technical`, `Other` |
| `WorkflowHolonId` | Guid | FK → CRE workflow engine (returned by `POST /api/workflow/save`) |
| `ParentSOPId` | Guid | FK → the SOPHolon this was forked from on STARNET |
| `SourcePublicOnSTARNET` | bool | Set true to publish on STARNET marketplace |
| `LicenceType` | Enum | `MIT` (free fork) / `Commercial` (treasury-gated) / `KarmaGated` (karma-score gated) |

**Lifecycle:**

```
Author defines SOP in SOPBuilder
    ↓
POST /api/workflow/save → WorkflowHolonId assigned
    ↓
SOPHolon.Status = "Active"
    ↓
Runs create SOPRunHolon records
    ↓
AI detects deviations → SOPVersionHolon (AIProposed)
    ↓
Human approves → Version incremented, old SOPVersionHolon archived
```

---

### SOPVersionHolon

An immutable snapshot of `SOPHolon` at a specific approved version. Created automatically when a version is incremented.

| Field | Type | Notes |
|-------|------|-------|
| `SOPId` | Guid | FK → parent SOPHolon |
| `Version` | string | The version captured |
| `SnapshotJson` | string | Full JSON of SOPHolon + all step definitions at this version |
| `Status` | Enum | `Approved` / `AIProposed` / `Rejected` |
| `ChangedByAvatarId` | Guid | Avatar who approved this version |

`AIProposed` versions are generated by the nightly BRAID improvement loop — clustering deviations and suggesting rewrites. They sit in a queue for human review before becoming `Approved`.

## StepZome

### SOPStepHolon

The atomic unit of an SOP. One holon = one instruction. Steps are ordered by `StepNumber`; equal step numbers run in parallel.

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `StepNumber` | int | Ordinal (1-indexed). Equal numbers = parallel execution. |
| `AssignedRole` | string | Role responsible for this step |
| `ConnectorType` | Enum | CRE connector — see table below |
| `ConnectorAction` | string | e.g. `create`, `update`, `sign`, `notify` |
| `AIPrompt` | string | BRAID prompt for this step. Supports `{input.*}` variable interpolation. |
| `RequiresAvatarSignOff` | bool | Cannot complete without wallet signature |
| `RequiresEvidenceUpload` | bool | Must upload at least one file before completing |
| `OnFailure` | Enum | `abort` / `skip` / `retry` / `escalate` |
| `TimeoutMinutes` | int | Deviation flagged if step exceeds this duration |

**Connector types available:**

| ConnectorType | Use in SOP |
|---------------|-----------|
| `holon` | Read or write a holon — lookup, create, update |
| `avatar` | Request Avatar sign-off / signature |
| `braid` | Send prompt to AI, receive analysis or draft |
| `webhook` | Call external system (CRM, Slack, Calendar, etc.) |
| `condition` | Evaluate a field value and branch |
| `trigger` | Fire a downstream event or SOP |
| `agent` | Delegate step to an autonomous A2A agent |
| `nft` | Issue or verify an NFT (e.g. certification, completion badge) |
| `wallet` | Check balance, release escrow, process payment |
| `karma` | Award karma to Avatar on step completion |
| `quest` | Trigger or complete a Quest linked to this SOP |
| `delay` | Pause execution for N minutes/hours (e.g. waiting period) |

---

### SOPDecisionPointHolon

A branching node. Reads an output field from the preceding step and routes to one of several `SOPConditionBranchHolon` paths.

| Field | Type | Notes |
|-------|------|-------|
| `ConditionField` | string | Output field to evaluate, e.g. `IntegrationComplexityScore` |
| `ConditionType` | Enum | `numeric_threshold` / `string_equals` / `bool_flag` / `ai_evaluated` |
| `DefaultBranchId` | Guid | Branch used if no explicit match |

`ai_evaluated` sends the field value to BRAID with context — useful for subjective decisions like "is this document sufficient?" or "does this scope qualify for the enterprise path?"

---

### SOPConditionBranchHolon

A named path from a decision point.

| Field | Type | Notes |
|-------|------|-------|
| `BranchName` | string | e.g. `ENTERPRISE PATH`, `GREEN`, `SIMPLE PATH` |
| `ConditionValue` | string | e.g. `>=3`, `RED`, `true` |
| `FirstStepId` | Guid | FK → SOPStepHolon where this branch begins |

## RunZome

### SOPRunHolon

One execution instance of an SOP. Created automatically when a trigger fires or a manual run is started.

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `WorkflowExecutionId` | string | The `executionId` from `POST /api/workflow/execute` |
| `Status` | Enum | `Running` / `Completed` / `Failed` / `Paused` / `Abandoned` |
| `InitiatedByAvatarId` | Guid | Who or what started this run |
| `AssignedAvatarIds` | string[] | All participants across all roles |
| `RunInputs` | string | JSON of trigger inputs |
| `RunOutputs` | string | Accumulated outputs as steps complete |
| `ProofHolonId` | Guid | Tamper-proof holon from `/api/workflow/verify` — set on completion |
| `ExternalContextId` | string | External record ID (e.g. Salesforce opportunity, Zendesk ticket) |

---

### StepCompletionHolon

Evidence record for one completed step within a run. The chain of `StepCompletionHolon` records across a `SOPRunHolon` forms the full audit trail.

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `CompletedByAvatarId` | Guid | Who completed this step |
| `StepInputs` / `StepOutputs` | string | Actual values used/produced |
| `AIGuidanceUsed` | bool | Did the executor consult BRAID? |
| `AIOutputSummary` | string | What BRAID said for this step |
| `EvidenceFileIds` | string[] | Uploaded files via FilesController |
| `EvidenceHash` | string | SHA-256 of all evidence — tamper detection |
| `AvatarSignature` | string | Wallet signature (if sign-off required) |
| `DurationSeconds` | int | Wall-clock time for this step |

---

### SOPDeviationHolon

Created when AI, timeout, or a human detects something wrong mid-run. Feeds the SOPIntel heatmap and the nightly AI improvement loop.

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `DeviationType` | Enum | `Timeout` / `MissingEvidence` / `SkippedRequired` / `AIAnomaly` / `ManualFlag` / `ExternalSystemFailure` |
| `Severity` | Enum | `Low` / `Medium` / `High` / `Critical` |
| `AIAnalysis` | string | BRAID's interpretation of root cause |
| `Status` | Enum | `Open` / `Acknowledged` / `Resolved` / `Escalated` |

`Critical` deviations can automatically trigger an escalation `SOPRunHolon` for a separate escalation procedure.

## AuditZome

### SOPAuditHolon

The tamper-proof audit certificate for a completed run. Created automatically and linked to a `ProofHolonId` verifiable on-chain via `GET /api/workflow/verify/{holonId}`.

**Use cases:**
- Regulatory compliance (FCA, SEC, GDPR)
- LP reporting for investment funds
- Contractual SLA evidence
- Insurance claim documentation
- Carbon credit verification

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `SOPVersion` | string | Exact SOP version executed |
| `StepsCompleted` | int | Steps successfully completed |
| `DeviationCount` / `CriticalDeviationCount` | int | Deviation summary |
| `AvatarSignatures` | string[] | All wallet signatures from the run |
| `EvidenceHashes` | string[] | All file evidence hashes |
| `ProofHolonId` | Guid | On-chain verifiable proof |

---

### SOPAnalyticsHolon

One record per SOPHolon. Updated after every completed run. Powers the SOPIntel dashboard.

**Key fields:**

| Field | Type | Notes |
|-------|------|-------|
| `CompletionRate` | double | % of runs reaching Completed |
| `AvgDurationSeconds` / `P95DurationSeconds` | double | Duration distribution |
| `DeviationsPerRun` | double | Mean deviation rate |
| `MostDeviatedStepId` | Guid | Step with highest deviation count — primary improvement target |
| `AIImprovementsPending` | int | Queued AI-proposed version upgrades |

## IntegrationZome

### IntegrationTriggerHolon

What fires a new `SOPRunHolon`.

| TriggerType | Example |
|-------------|---------|
| `Webhook` | Salesforce fires when deal stage = "Closed Won" |
| `Schedule` | Monthly ESG report SOP fires on 1st of month |
| `Manual` | User clicks "Start Run" in SOPRunner |
| `HolonEvent` | `ContainerHolon.Status` changes to `GateIn` |
| `SOPRunComplete` | Previous SOP run completing triggers next one |

---

### WebhookConfigHolon

One record per step-system connection. Configured inline in SOPBuilder.

**Supported systems:** Salesforce, HubSpot, Zendesk, Freshdesk, Slack, Microsoft Teams, Google Calendar, Outlook, DocuSign, HelloSign, Zapier, Make, Jira, ServiceNow, Custom.

**PayloadTemplate** uses `{input.*}` and `{output.*}` variable interpolation, allowing step outputs to be passed directly into external system API calls.

---

### ExternalSystemLinkHolon

Bi-directional lookup record between a `SOPRunHolon` and an external system record.

| Field | Notes |
|-------|-------|
| `SystemName` | e.g. "Salesforce" |
| `RecordType` | e.g. "Opportunity", "Ticket" |
| `ExternalId` | The record's ID in the external system |
| `ExternalUrl` | Direct deep link to the record |


## Why Holonic SOPs Are Uniquely Powerful

A traditional SOP lives in a document. It cannot be run. It cannot detect when it is not being followed. It cannot learn from deviations. It cannot be verified by a third party. When an employee follows it, there is no record. When they deviate, nobody knows. When a regulator asks for proof of compliance, someone assembles a spreadsheet.

A holonic SOP is a different kind of object entirely. The `SOPHolon` is not a document — it is a typed, versioned, Avatar-owned, blockchain-anchored definition that every execution instance references. Every `SOPRunHolon` is an immutable record of exactly what happened, who did it, when, and with what evidence. Every `StepCompletionHolon` is a signed receipt. Every `SOPDeviationHolon` is a timestamped anomaly that the system detected, not a human after the fact.

The AI layer does not replace human judgement — it amplifies it. BRAID drafts the SOP from a conversation. It guides the executor at each step with context that no checklist can provide. It reads every run, clusters deviations by step, and generates specific rewrite suggestions — not "this process could be better" but "Step 3a has a 34% timeout rate; the TimeoutMinutes should be increased from 60 to 120 and the step description should include the API key setup instructions that are currently buried in Step 7." And because every SOPVersionHolon is immutable, the history of how a procedure evolved — and why — is permanently queryable.

The STARNET dimension makes this a platform, not just a tool. When a successful consultancy like Bryan's publishes their customer onboarding SOP — battle-tested across 800 companies — other organisations do not copy a Word document. They fork a holonic blueprint: every step, every decision point, every webhook integration, every AI prompt, pre-configured. They customise it for their context, run it, improve it, and optionally publish their improved version back. The entire ecosystem of business operations becomes composable, auditable, and AI-improvable — and the Karma system ensures that the most effective SOPs rise to the top.

No other platform has this. Notion, Confluence, Trainual, Process.st, Tallyfy — none of them have canonical identity across storage providers, on-chain proof, Avatar accountability, AI-native design, and a global marketplace for reusable processes. The holon is not the container — the holon is the procedure itself.
